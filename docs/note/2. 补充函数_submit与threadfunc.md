# 完成SubmitTask函数

## 1. 条件变量的三种等待方式

在 `std::condition_variable` 中，等待机制有三种，区别主要在于“等待多久”：

| 方法             | 说明                                                         | 使用场景                                                     |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`wait`**       | **无限期等待**。直到另一个线程调用 `notify` 且条件满足为止。 | 任务必须执行，不计时间成本。                                 |
| **`wait_for`**   | **限时等待（时长）**。如 `wait_for(lock, 1s)`。如果在 1s 内条件满足则返回 `true`，否则返回 `false`。 | **防止死等**。如用户提交任务时，如果不希望程序卡死，可设置超时。 |
| **`wait_until`** | **限时等待（时刻）**。如等待直到“今天 24:00”。               | 需要在特定时间点执行或停止的场景。                           |

---

## 2. 等待条件满足的两种书写方式

在多线程中，由于存在**虚假唤醒（Spurious Wakeup）**（即线程在没有被 notify 的情况下也可能醒来），必须在唤醒后重新检查条件。

### 方式 A：手动 while 循环
```cpp
while(taskQue_.size() == taskQueMaxThreshHold_) {
    notFull_.wait(lock);
}
```
*   **原理**：线程醒来后回到 `while` 判断条件。如果条件仍不满足（队列还是满的），继续等待。

### 方式 B：使用 Lambda 表达式（推荐）
```cpp
notFull_.wait(lock, [&]() { 
    return taskQue_.size() < taskQueMaxThreshHold_; 
});
```
*   **原理**：这是 C++11 提供的语法糖。其内部逻辑等同于方式 A。只有当 Lambda 返回 `true` 时，`wait` 才会结束并继续向下执行。

---

## 3. 唤醒机制：notify_one vs notify_all

当生产者放入任务或消费者取走任务后，需要通知等待中的线程：

*   **`notify_one()`**：
    *   **动作**：只唤醒**一个**正在等待该条件变量的线程。
    *   **优点**：效率高，避免“惊群效应”（Thundering Herd）。
    *   **适用**：所有等待线程处理逻辑相同，且一次通知只能由一个线程处理（如：加入一个任务，只需唤醒一个工人线程）。

*   **`notify_all()`**：
    *   **动作**：唤醒**所有**正在等待该条件变量的线程。
    *   **适用**：
        1. 存在多种类型的等待者（如读写锁）。
        2. 状态发生重大变化（如线程池销毁，需要唤醒所有线程退出）。
        3. 代码中 `notEmpty_.notify_all()` 是安全的，但在高并发下，如果有大量空闲线程，`notify_one` 可能更优。

---

## 4. 为什么使用 unique_lock？

在 `submitTask` 中，必须使用 `std::unique_lock` 而不是 `std::lock_guard`：
1.  **自动释放与重新获取**：当执行 `wait` 时，`unique_lock` 会自动**释放锁**，让其他线程可以访问队列；当被唤醒时，它会自动**重新获取锁**。
2.  **灵活性**：`unique_lock` 支持手动 `unlock` 和 `lock`，而 `wait` 机制必须要求锁具备这种灵活性。

*   **Q: 为什么 wait 必须放在锁的范围内？**
    *   A: 为了保证“判断条件”和“进入等待状态”这两个动作是原子的。否则可能判断条件时还没满，准备等待时别人已经满了，导致信号丢失。
*   **Q: wait 阻塞时会占用 CPU 吗？**
    *   A: 不会。线程会进入等待队列并挂起



# 完成ThreadFunc函数

## 1. 核心逻辑流程
`ThreadFunc` 是线程池中每个工作线程的入口函数，其核心步骤为：

1. **争抢锁**：所有空闲线程竞争 `taskQueMtx_`。
2. **等待任务**：如果队列为空，通过 `notEmpty_.wait()` 挂起并释放锁。
3. **取出任务**：一旦被唤醒且队列有任务，取出任务并更新队列状态。
4. **通知生产**：任务取出后，队列有了新空位，通知生产者（`notFull_.notify`）。
5. **执行任务**：**释放锁后**再执行任务，确保不阻塞其他线程取任务。

---

## 2. 关键知识点深度解析

### ① 精确的锁作用域 (Lock Scoping)
代码中使用了大括号 `{ ... }` 显式限定了 `std::unique_lock` 的生命周期：
*   **原因**：执行任务（`task->run()`）往往非常耗时。如果在执行任务时还持有锁，那么其他线程都无法进入队列取任务，线程池将退化为“单线程串行执行”。
*   **原则**：**锁的粒度越小越好**。只在操作共享资源（队列）时持锁，一旦数据到手，立即解锁。

### ② 条件变量的“双向通信”
*   **`notEmpty_.wait(lock, ...)`**：消费者在此阻塞，等待生产者通知“有米下锅了”。
*   **`notFull_.notify_all()`**：消费者取走任务后，通知生产者“有空位了，可以继续生产”。
*   **关于代码中的 `if(taskQue_.size()>0) notEmpty_.notify_all();`**：
    *   这是一个优化技巧（链式通知）。当一个线程被唤醒并取走一个任务后，如果发现队列里还有剩余任务，它顺便通知其他正在等待的同伴起来干活。

### ③ 智能指针的生命周期
*   在循环外定义 `std::shared_ptr<Task> task;`，在锁内赋值。
*   这样可以确保 `Task` 对象的引用计数在执行期间至少为 1，防止任务在执行过程中被意外销毁。

### ④ 优雅停机的隐患 
当前使用了 `for(;;)` 死循环。在实际工业级开发中，必须考虑**如何让线程停下来**：
*   通常需要一个 `std::atomic<bool> isRunning_` 标志。
*   当线程池析构时，将标志设为 `false`，并 `notify_all` 所有线程。
*   线程醒来后检查标志，若是 `false` 则退出循环。

---

*   **Q: 为什么取任务要加锁，执行任务不加锁？**
    *   A: 任务队列是共享资源，多线程同时 `pop` 会导致内存错误（Race Condition）；而任务执行通常是独立的，不涉及共享资源修改，锁外执行可以实现真正的并行。
*   **Q: 这里用 `notify_one` 还是 `notify_all`？**
    *   A: 在 `submitTask`（生产者）中，建议用 `notify_one` 唤醒一个工人即可。在 `ThreadPool` 析构时，必须用 `notify_all` 唤醒所有工人准备下班。
*   **Q: `wait` 内部做了什么？**
    *   A: 1. 释放 `mutex`。 2. 线程进入等待状态。 3. 被唤醒时，重新尝试获取 `mutex`。 4. 获取成功后，检查 Lambda 条件。
